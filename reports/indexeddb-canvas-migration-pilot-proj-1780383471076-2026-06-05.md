# IndexedDB 画布状态迁移试点结果

试点项目：`proj-1780383471076`

测试方式：关闭正式 App 后，使用 APFS 写时复制建立完整正式 userData 副本；实验源码以正式端口 `54134` 和独立 userData 路径启动，只修改副本。

## 迁移结果

- 迁移前画布状态：3 个节点、2 条连线，JSON 序列化约 1,303,711 字节。
- 迁移后画布状态：3 个节点、2 条连线，JSON 序列化约 15,954 字节。
- 冷启动重新读取：3 个节点、2 条连线，JSON 序列化约 16,066 字节。
- `portableDataRef` 从 4 个减少到 3 个；被移除的是已具备本地文件绑定的视频内嵌副本。
- 节点 ID 与连线迁移前后保持一致。

## 应用验证

- 迁移后重新加载页面，显示 3 个节点和 2 条连线。
- 两段视频均成功加载，`readyState = 4`，无媒体错误。
- 一段音频成功加载，`readyState = 4`，无媒体错误。
- 从迁移项目切换到 `测试A`，再切回迁移项目，节点、连线和媒体均正常。
- 关闭实验进程后冷启动，迁移状态仍正常恢复。

## 空间变化

- 副本 IndexedDB 文件总字节数：
  - 迁移前：8,642,238,356 字节。
  - 迁移后：8,638,892,348 字节。
  - 立即减少：3,346,008 字节，约 3.19 MiB。
- 单项目状态显著缩小，但 LevelDB 不会立即按逻辑删除量等比例释放物理空间。

## 发现的限制

- 项目此前已经部分外部化，迁移后仍保留 6 处指向正式媒体库或工具输出目录的文件引用。
- 当前迁移逻辑会复用已有 `file://` 文件绑定，不会把这些文件重新复制到新的迁移目录。
- 因此当前方案可以明显减轻 IndexedDB 压力，但还不能保证项目脱离原媒体库后独立可用。
- 剩余 3 个 `portableDataRef` 均属于文本类内容，不是图片、视频或音频内嵌值。

## 正式数据安全

- 正式 IndexedDB 修改时间未变化。
- 正式试点媒体目录修改时间未变化。
- 正式源码仓库保持干净。
