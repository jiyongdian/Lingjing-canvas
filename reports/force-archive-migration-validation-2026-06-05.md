# 强制归档迁移验证报告

测试范围：只修改实验源码和正式 userData 的 APFS 写时复制副本；正式 App 数据、正式媒体库和正式源码均未修改。

## 实现

- `prepareProjectMediaStateForPersistence` 新增 `forceRehomeExistingFiles` 模式。
- 已有 `file://`、绑定 `localPath` 和节点数据中的嵌套文件引用会重新归档到指定媒体库。
- 归档文件按 SHA-256 内容寻址，同一项目内相同内容只保存一次。
- 本地文件归档使用分块哈希和文件复制，避免大视频一次性读入内存。
- 强制归档会清除媒体绑定中的 `portableDataRef`、`portableData` 和旧路径值。
- 默认模式保持不变，只有显式开启强制归档时才重迁已有文件。

## 渲染压力对比

三个项目迁移并冷启动压缩后，测试副本 IndexedDB 从 8,661,600,460 字节降到 8,426,386,365 字节，物理减少约 224.32 MiB。

大型项目 `椰子水-6.5`：

- 冷启动加载时间：约 697 ms 降到 568 ms，改善约 18.5%。
- 3 秒采样 FPS：约 40.0 提升到 43.3，改善约 8.3%。
- JS 堆：约 500 MB 降到 358 MB，降低约 28%。
- 冷启动后 16 节点、15 连线全部恢复，视频错误 0，音频错误 0。

中小项目 FPS 变化在正常测量波动范围内。强制归档显著降低大状态解析和内存压力，但视频解码、节点组件和连线渲染仍然是主要压力来源。

## 新项目生成链路

- 新生成图片首次保存进入 `项目ID/blobs/SHA-256.png`。
- 第二次保存相同结果返回 `deduplicated: true`。
- 两次保存路径完全相同，没有产生第二个物理副本。
- `contentAddressed: true`，并且路径位于指定归档媒体库。

## 测试中发现并修复的问题

- 未重新构建 `dist` 时测试进程仍运行旧逻辑；已修正测试流程。
- 打开中的项目可能由旧 React 状态自动保存并覆盖后台迁移结果。正式实现迁移 UI 时，必须先切换到安全项目或同步更新当前画布状态。
- 首版只迁移绑定路径，未迁移节点数据中的嵌套 `file://`；已增加递归归档。
- 首版文本绑定的冗余 `binding.value` 仍保留旧路径；已统一改写并清除便携副本。

## 剩余限制

- 当前内容寻址去重范围是项目内。不同项目包含相同媒体时，仍会各保存一份。
- 强制归档模式目前是内部迁移能力，尚未增加面向用户的迁移按钮、进度界面和两阶段提交 UI。
- 正式启用前需要实现当前项目迁移锁、失败回滚清单和跨项目全局 Blob 存储。
