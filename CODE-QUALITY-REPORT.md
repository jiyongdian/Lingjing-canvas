## 问题一：主题色"换了不生效"（高优先级，已规划修复）

### 现状：两套割裂的样式世界

应用其实**已有一套完整的主题 token 系统**，但只有一半组件接入：

- **已接入主题（好）**：`electron/main/ui-overrides.css`（运行时注入）定义了 **72 个 `--wanjuan-*` 语义 token**，通过 `html.theme-X { --wanjuan-theme-bg: #hex }` → `--wanjuan-bg: var(...)` → `var(--wanjuan-bg)` 的级联，换 `theme-X` class 即可整体换肤。架构本身是正确的。
- **未接入主题（坏）**：
  - `src/renderer/styles/app.css`（68KB 主样式）：**仅 1 处** `var()`，**248 处硬编码 `#hex`**
  - 前端 JS `index.js`：**Tailwind 硬编码色 1336 处**（如 `bg-[#222]`、`border-[#333]`）+ **内联 style 硬编码色 997 处**

### 为什么换主题色不生效

主题 token 改了，但 app 绝大多数颜色写死在 JS 的 Tailwind 类名里（`border-[#333]` 出现 **313 次**、`bg-[#222]` **126 次**…），**这些不读 token**，所以纹丝不动。

### 高频硬编码色 → 应映射的 token

| 硬编码 | 次数 | 语义 | 目标 token |
|--------|------|------|-----------|
| `border-[#333]` | 313 | 边框 | `--wanjuan-border` |
| `bg-[#222]` | 126 | 面板 | `--wanjuan-panel` |
| `bg-[#333]` | 104 | 控件底 | `--wanjuan-control-bg` |
| `bg-[#2a2a2a]` | 102 | 次级面板 | `--wanjuan-panel-2` |
| `bg-[#121212]` | 98 | 画布底 | `--wanjuan-bg` |
| `bg-[#1a1a1a]` | 89 | 深面板 | `--wanjuan-bg` |
| `bg-[#1c1c1c]` 等 | 各数十次 | 面板系 | 同上 |

不同的硬编码灰度值约 20 余种，集中度高，可批量映射。

## 问题二："改一处坏一处"（结构性，高风险治理）

巨型"上帝组件"：

| 组件 | 行数 | useState 数 |
|------|------|------------|
| `St`（根组件） | 19588 | **168** |
| `dt` | 11955 | 13 |
| `Le` | 7685 | — |

`St` 单个函数 1.96 万行、168 个状态全挤在一个作用域。任何改动都可能牵连其他状态，无边界隔离——这是"说哪里改哪里、改完别处坏"的结构根源。（连线 bug 就发生在这类巨块的重命名中。）

## 问题三：配置散落 + 逻辑重复（中优先级）

- **模型/API 配置无集中层**：模型名（`claude-`/`gpt-`/`gemini-`）硬编码 **254 处**，API 地址 `s.lconai.com` 7 处。改模型列表要全文件找。
- **网络/存储逻辑重复**：`fetch(` **62 处**、"请求失败"错误处理 **41 处**、`localStorage` 直接读写 **31 处**。同样的模式反复手写，无统一封装。

## 治理建议（按"收益/风险"排序）

1. **修主题系统**（已规划下一步做）：把 app.css + JS 高频硬编码色接入现有 token。让换主题色真正生效。中等风险，逐步验证。
2. **抽配置 + 封装公共层**：模型列表/API 地址集中化，统一 fetch/storage 封装。中等风险。
3. **拆巨型组件**（最高风险）：把 `St`/`dt` 拆为子组件。收益最大但需重构 168 个状态，极易静默损坏，建议最后做且充分回归。

## 修复记录：主题系统（2026-05-30 已完成）

### 重要更正
审查时初判"主题色完全不生效"过于武断。实测发现：原作者在 `ui-overrides.css` 里已用属性选择器
（`html:is(.theme-X) [class*="bg-[#222]"] { background: var(--token) }`）做了**主题覆盖劫持**，
把硬编码 Tailwind 色重定向到主题 token。切换主题**大部分生效**——覆盖了 1338 次中的 1118 次（84%）。

### 真实问题与修复
原覆盖规则只处理了 15 种高频色，**95 种低频杂色（220 次，16%）漏网**，切主题时不变，
造成局部色块突兀（浅色主题下尤其明显）。

修复：按语义（深色 bg→surface、border→border token、浅色 text→text token）为漏网色
**补充覆盖规则**，追加到 `ui-overrides.css`。**仅新增 CSS，零 JS 改动，与原机制一致，零破坏风险。**

结果：主题覆盖率 **84% → 98.8%**。剩余 9 种是品牌强调色/特定语义色，本就不应跟随主题，保留正确。
已实测验证（`#23262d` 等漏网色现可随 graphite↔warm-light 正确切换）。
